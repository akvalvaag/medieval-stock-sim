# Feature Expansion Design ŌĆö Medieval Market Simulator
**Date:** 2026-03-23
**Status:** Approved

---

## Overview

Add 8 new features to the medieval stock market simulator: Seasons, NPC Bots, Limit Orders, Moneylender, Warehousing Costs, Price Alerts, Achievements, and Net Worth Chart.

Architecture: **Service Pipeline**. Each new concern is a focused `@Component` with a `processTick()` method. `MarketEngine` orchestrates them in sequence each tick.

---

## Backend

### Tick Order (MarketEngine)
1. `seasonEngine.advanceTick()` ŌĆö roll season, get modifiers
2. Reprice all goods (existing logic + season nudge)
3. `botService.processTick()` ŌĆö bots trade
4. `limitOrderService.processAll(humanPortfolios)` ŌĆö trigger pending orders, collect fills
5. `moneylenderService.processAll(humanPortfolios)` ŌĆö compound interest + default seizure
6. `warehousingService.processAll(humanPortfolios)` ŌĆö deduct holding fees
7. Broadcast global `MarketState` to `/topic/market` (season + goods + scoreboard + event)
8. For each human session: push `SessionUpdate` to `/queue/updates` via `convertAndSendToUser`

**Bot portfolios are excluded from steps 4ŌĆō6 and step 8.** `Portfolio` gains a `boolean bot` flag. `SessionRegistry.getHumanPortfolios()` filters on `!bot`. Bots are also **exempt from expiry** ŌĆö the 2-hour inactivity check skips portfolios where `bot=true`.

### WebSocket Config Changes

Two required changes to `WebSocketConfig`:

1. **Enable `/queue` broker:**
   ```java
   config.enableSimpleBroker("/topic", "/queue");
   ```

2. **Register user destination prefix** (already needed for `convertAndSendToUser`):
   ```java
   config.setUserDestinationPrefix("/user");
   ```

3. **Principal resolution:** Add a `HandshakeInterceptor` that reads the `sessionId` query parameter from the SockJS connection URL (`ws://ŌĆ”/ws?sessionId=xxx`) and wraps it as a `Principal`. Spring then uses this Principal to route `/queue/updates` to the correct STOMP session. The interceptor sets the `Principal` in the `WebSocketSession` attributes map; a custom `DefaultHandshakeHandler` subclass reads it from the attributes and returns it as the user Principal.

Frontend connects via:
```js
const socket = new SockJS('/ws?sessionId=' + session.id);
```
Frontend subscribes to:
```js
stompClient.subscribe('/user/queue/updates', handler);
```

### SeasonEngine

**Responsibility:** Track a 4-season cycle; expose passive per-tick drift nudges.

- Each season = 60 ticks (~5 min at 5s/tick)
- Cycle: SPRING ŌåÆ SUMMER ŌåÆ AUTUMN ŌåÆ WINTER ŌåÆ repeat
- `advanceTick()` increments counter; rolls season at 60; returns current season
- `getModifiers()` returns `Map<String, Double>` of per-tick % nudge for affected goods

**Season modifiers (applied as additive drift each tick):**

| Season | Boosts | Suppresses |
|--------|--------|------------|
| ­¤ī▒ Spring | Grain +8%, Wool +8%, Livestock +8%, Fish +5% | Coal ŌłÆ10%, Candles ŌłÆ10% |
| ŌśĆ’ĖÅ Summer | Spices +10%, Herbs +12%, Ale +5%, Cloth +8% | Grain ŌłÆ10% |
| ­¤Źé Autumn | Grain +8%, Timber +10%, Ale +10%, Coal +8%, Leather +5% | ŌĆö |
| ŌØä’ĖÅ Winter | Coal +20%, Candles +15%, Salt +10%, Spices +8%, Grain +12% | Fish ŌłÆ10%, Livestock ŌłÆ8% |

Season modifiers are scaled down to a small per-tick nudge (divide by 100) so they accumulate gradually rather than shocking prices.

**`MarketState` additions:** `season` (String), `seasonTicksRemaining` (int).

### BotService

**Responsibility:** Simulate 5 NPC traders who affect prices and appear on the scoreboard.

- Bots created at startup; registered in `SessionRegistry` with `Portfolio(bot=true)` and names prefixed `[bot]`
- Starting gold: 300g
- Bot names generated by `NameGenerator` with `[bot] ` prefix
- Every 3 ticks, each bot picks a random good and decides:
  - `currentPrice < 80% of basePrice` AND has gold ŌåÆ buy 1ŌĆō5 units
  - `currentPrice > 120% of basePrice` AND holds some ŌåÆ sell 1ŌĆō5 units
  - Otherwise ŌåÆ 50/50 buy or sell 1ŌĆō3 units (if affordable / has holdings)
- Bots use `TradeService` (fees apply, supply pressure applies)
- Bots appear on scoreboard; no WebSocket connection needed
- Bots are **excluded** from warehousing fees, moneylender, expiry, and per-session pushes

### LimitOrderService

**Responsibility:** Execute deferred buy/sell orders when price conditions are met.

**Model: `LimitOrder`**
```
id          UUID (generated on creation)
sessionId   String (owner ŌĆö for DELETE lookup and per-session broadcast)
goodName    String
direction   BUY | SELL
quantity    int
targetPrice double
```

- BUY triggers when `currentPrice Ōēż targetPrice`
- SELL triggers when `currentPrice Ōēź targetPrice`
- Max 3 pending orders per portfolio
- Orders stored as `List<LimitOrder>` in `Portfolio`
- `processAll(humanPortfolios)` iterates each portfolio's orders each tick:
  - For sell orders: read `avgCostBasis` from portfolio **before** calling `TradeService.sell()`, then compute `realizedPnl = (currentPrice ŌłÆ avgCostBasis) ├Ś qty ├Ś (1 ŌłÆ feeRate)`
  - Execute via `TradeService`; if throws ŌåÆ cancel order silently
  - Triggered/cancelled orders removed from portfolio
- After processing, the updated order list + any fill events are included in the per-session `SessionUpdate` push

**New endpoints:**
- `POST /api/orders` ŌĆö body: `{sessionId, goodName, direction, quantity, targetPrice}` ŌåÆ `OrderResponse{orders, gold}`
- `DELETE /api/orders/{id}?sessionId={sessionId}` ŌĆö sessionId as query param ŌåÆ `OrderResponse{orders, gold}`

### MoneylenderService

**Responsibility:** Allow players to borrow gold at compounding interest; seize assets on default.

- Loan tracked in `Portfolio`: `double loanAmount` (default 0.0)
- Borrow limit: `min(2 ├Ś currentGold, 5000g)`; minimum borrow: 50g
- If `2 ├Ś currentGold < 50g`, the borrow button is disabled (max < minimum); the `POST /api/borrow` endpoint rejects with an error if `amount < 50` or `amount > limit`
- Interest: **2% per tick** (compounding)
- Each tick (human portfolios only):
  1. If `loanAmount > 0`: `loanAmount *= 1.02`
  2. Check solvency: if `loanAmount > gold + holdingsValue` ŌåÆ seize
  3. Seizure: sort holdings by `currentPrice ├Ś qty` ascending; for each holding, snapshot `goldBefore = portfolio.getGold()`, call `TradeService.sell()`, compute `proceeds = portfolio.getGold() - goldBefore`, reduce `loanAmount -= proceeds`; stop when `loanAmount Ōēż 0`. Any remaining negative `loanAmount` is forgiven (set to 0).
- Repay (`POST /api/repay`): always repays the **maximum possible amount** ŌĆö deducts `min(gold, loanAmount)` from gold and from `loanAmount`; clears loan if fully repaid. No `amount` parameter.

**New endpoints (sessionId in request body):**
- `POST /api/borrow` ŌĆö `{sessionId, amount}` ŌåÆ `LoanResponse{loanAmount, gold}`
- `POST /api/repay` ŌĆö `{sessionId}` ŌåÆ `LoanResponse{loanAmount, gold}`

**Portfolio additions:** `double loanAmount` (default 0.0).

### WarehousingService

**Responsibility:** Charge a small per-tick fee on held inventory to discourage passive hoarding.

- Applies to **human portfolios only** (bots excluded)
- Fee rates added to `PlayerClass` enum as a field `double warehousingRate`:
  - **MERCHANT**: 0.0025 (0.25%)
  - **MINER**: 0.005 (0.5%)
  - **NOBLE**: 0.005 (0.5%)
- `holdingsValue = ╬Ż (qty ├Ś currentPrice)` for all held goods
- Fee = `holdingsValue ├Ś warehousingRate`
- Deduction skipped if gold would go below 0
- Silent ŌĆö no notification to client

### TradeService Changes

The controller layer (not `TradeService.sell()` itself) must compute `realizedPnl` for sell trades using a **gold-delta approach** to automatically capture all bonuses (including the MINER Mining-category +2% bonus):

1. **Before** calling `tradeService.sell()`:
   - Read `avgCostBasis = portfolio.getAvgCostBasis(goodName)` (cleared by `sell()` on full sells)
   - Read `goldBefore = portfolio.getGold()`
2. Call `tradeService.sell(portfolio, goodName, quantity)`
3. Compute:
   - `saleProceeds = portfolio.getGold() - goldBefore`
   - `originalCost = avgCostBasis * quantity`
   - `realizedPnl = saleProceeds - originalCost`

This approach captures all sell-path modifiers (fees, MINER bonus) without duplicating `TradeService` logic.

`TradeResponse` gets a new field `double realizedPnl` (0.0 for buy operations).

## WebSocket Channels

### Global broadcast: `/topic/market`
Sent to all subscribers each tick:
```json
{
  "goods": [...],
  "scoreboard": [...],
  "season": "WINTER",
  "seasonTicksRemaining": 38,
  "eventMessage": "..."
}
```

### Per-session push: `SessionUpdate` to `/queue/updates`
Sent via `convertAndSendToUser(sessionId, "/queue/updates", payload)` after each tick:
```json
{
  "gold": 842.0,
  "limitOrders": [...],
  "loanAmount": 250.0,
  "limitOrderFills": [{ "goodName": "Iron", "direction": "SELL", "quantity": 5, "executedPrice": 65.0, "realizedPnl": 45.0 }]
}
```
`gold` is included so the frontend refreshes the displayed gold balance each tick (warehousing fees and moneylender interest are deducted server-side each tick without a trade, so `TradeResponse` alone is insufficient to keep gold current).

`limitOrderFills` is a list of orders that executed this tick (empty list if none). Includes `realizedPnl` so the frontend can use it for achievement tracking (e.g., `seasonal_trader`).

## Data Flow Changes

### `MarketState` (global `/topic/market` broadcast)
Add: `season` (String), `seasonTicksRemaining` (int). `eventMessage` already present.

### `SessionUpdate` (new ŌĆö per-session `/user/queue/updates`)
```json
{ "gold": double, "limitOrders": [...], "loanAmount": double, "limitOrderFills": [...] }
```

### `SessionStartResponse`
Add: `season` (String), `seasonTicksRemaining` (int), `loanAmount` (double), `limitOrders` (List).

### `TradeResponse`
Add: `realizedPnl` (double ŌĆö 0.0 for buys; computed by controller before calling `sell()`), `loanAmount` (double), `currentSeason` (String).

### New DTOs
- `OrderResponse { List<LimitOrder> orders, double gold }`
- `LoanResponse { double loanAmount, double gold }`
- `LimitOrderFill { String goodName, String direction, int quantity, double executedPrice, double realizedPnl }`

### `PlayerClass` enum
Add field `double warehousingRate`:
- MERCHANT: 0.0025
- MINER: 0.005
- NOBLE: 0.005

## Out of Scope

- Achievements persisted server-side (localStorage only, per user decision)
- Price alert persistence across page refresh (session-only Vue state)
- Bot difficulty settings
- Seasonal events (separate from the random event engine)
- Partial loan repayment (repay always pays the maximum possible)
- Limit order fills contributing to `seasonal_trader` via a separate mechanism ŌĆö fills are included in `SessionUpdate.limitOrderFills` with `realizedPnl` and the current season is inferred from the last received `MarketState.season`
